Volume Migration: From ZAPI to REST API

Community Submission • Dec 8, 2024 • 6 min read

NetApp deprecated ZAPI (ONTAP API) in ONTAP 9.12.1 and it will be removed in a future release. If you're still using ZAPI-based scripts, now is the time to migrate to the modern REST API. This guide walks you through the key differences and how to rewrite your volume management scripts.

Why ZAPI is Going Away

ZAPI has been around since the early days of ONTAP and served us well. But REST APIs are the industry standard now — they're easier to use, better documented, and work with modern tooling like Ansible, Terraform, and Python's requests library.

Key Differences

  • Protocol: ZAPI uses XML over HTTP. REST uses JSON over HTTPS.
  • Authentication: Both support basic auth, but REST also supports OAuth 2.0.
  • Versioning: REST API is versioned (/api/v1/), making upgrades safer.
  • Discovery: REST API has self-documenting endpoints via Swagger UI at /docs/api.

Before: ZAPI Volume Create (Python)

Here's how you used to create a volume using the old NetApp Manageability SDK:

python 
# Old way - using ZAPI via NetApp Manageability SDK
from NaServer import NaServer, NaElement

def create_volume_zapi(cluster, user, password, vserver, vol_name, aggr, size):
    s = NaServer(cluster, 1, 21)
    s.set_server_type("FILER")
    s.set_transport_type("HTTPS")
    s.set_port(443)
    s.set_style("LOGIN")
    s.set_admin_user(user, password)
    s.set_vserver(vserver)

    api = NaElement("volume-create")
    api.child_add_string("volume", vol_name)
    api.child_add_string("containing-aggr-name", aggr)
    api.child_add_string("size", size)
    api.child_add_string("space-reserve", "none")
    api.child_add_string("junction-path", f"/{vol_name}")

    result = s.invoke_elem(api)
    if result.results_status() == "failed":
        raise Exception(result.results_reason())

    print(f"Volume {vol_name} created successfully")

# Usage
create_volume_zapi("192.168.1.100", "admin", "password", 
                   "svm1", "vol_test", "aggr1", "10g")

After: REST API Volume Create (Python)

Here's the modern equivalent using the ONTAP REST API:

python 
# New way - using ONTAP REST API
import requests
import urllib3
urllib3.disable_warnings()

def create_volume_rest(cluster, user, password, vserver, vol_name, aggr, size_mb):
    base_url = f"https://{cluster}/api"
    auth = (user, password)
    headers = {"Content-Type": "application/json"}

    payload = {
        "name": vol_name,
        "svm": {"name": vserver},
        "aggregates": [{"name": aggr}],
        "size": size_mb * 1024 * 1024,  # Convert MB to bytes
        "space": {"guarantee": {"type": "none"}},
        "nas": {"path": f"/{vol_name}"}
    }

    response = requests.post(
        f"{base_url}/storage/volumes",
        auth=auth,
        headers=headers,
        json=payload,
        verify=False
    )

    if response.status_code == 201:
        print(f"✅ Volume '{vol_name}' created successfully")
    else:
        print(f"❌ Error: {response.json()}")
        response.raise_for_status()

# Usage
create_volume_rest("192.168.1.100", "admin", "password",
                   "svm1", "vol_test", "aggr1", 10240)

Using the Official netapp-ontap Library

Even cleaner — use NetApp's official Python library which wraps the REST API:

bash 
pip install netapp-ontap
python 
from netapp_ontap import HostConnection, NetAppRestError
from netapp_ontap.resources import Volume

with HostConnection("192.168.1.100", username="admin", password="password", verify=False):
    volume = Volume(
        name="vol_test",
        svm={"name": "svm1"},
        aggregates=[{"name": "aggr1"}],
        size=10 * 1024 ** 3,  # 10 GB in bytes
        nas={"path": "/vol_test"}
    )

    try:
        volume.post()
        print(f"✅ Volume '{volume.name}' created")
    except NetAppRestError as e:
        print(f"❌ Error: {e}")

ZAPI → REST API Cheat Sheet

text 
ZAPI Call                        REST API Endpoint
─────────────────────────────────────────────────────
volume-create                 →  POST   /api/storage/volumes
volume-destroy                →  DELETE /api/storage/volumes/{uuid}
volume-get-iter               →  GET    /api/storage/volumes
volume-modify                 →  PATCH  /api/storage/volumes/{uuid}
snapshot-create               →  POST   /api/storage/volumes/{uuid}/snapshots
snapmirror-get-iter           →  GET    /api/snapmirror/relationships
vserver-get-iter              →  GET    /api/svm/svms
net-interface-get-iter        →  GET    /api/network/ip/interfaces

Common Pitfalls

  • Size units: ZAPI accepts strings like "10g". REST API expects bytes as an integer.
  • Async jobs: Some REST calls return a 202 Accepted with a job ID — you need to poll /api/cluster/jobs/{uuid} for completion.
  • Field names: ZAPI used hyphenated names (space-reserve). REST uses nested JSON objects.
  • SVM context: In ZAPI you set the vserver on the connection. In REST, it's part of the request body.

Handling Async Jobs

Some operations like volume creation return a job. Here's how to handle it:

python 
import time
import requests

def wait_for_job(cluster, auth, job_url, timeout=60):
    """Poll a job until it completes or times out."""
    start = time.time()
    
    while time.time() - start < timeout:
        response = requests.get(
            f"https://{cluster}{job_url}",
            auth=auth,
            verify=False
        )
        job = response.json()
        state = job.get("state")

        if state == "success":
            print("✅ Job completed successfully")
            return True
        elif state == "failure":
            print(f"❌ Job failed: {job.get('message')}")
            return False
        
        time.sleep(2)  # Wait 2 seconds before polling again

    print("⚠️ Job timed out")
    return False

# Usage after a POST that returns 202
response = requests.post(...)
if response.status_code == 202:
    job_url = response.json()["job"]["_links"]["self"]["href"]
    wait_for_job("192.168.1.100", auth, job_url)

Conclusion

Migrating from ZAPI to REST is a one-time effort that pays off. Your scripts become cleaner, easier to maintain, and future-proof. Start by identifying all ZAPI calls in your codebase, map them to REST equivalents using the cheat sheet above, and test thoroughly in a non-production environment first.
